# Building a token-lean Rust MCP server to replace Read, Grep, and Glob

**Bottom line up front:** build a Rust MCP server around `rmcp 1.3+`, tree-sitter with Aider-style repo-mapping as the default context, Tantivy + a custom identifier tokenizer for exact/fuzzy symbol search, SQLite (FTS5 trigram) for the canonical symbol table, LanceDB + a local ONNX code-embedding model for optional semantic fallback, and a Merkle + `notify` incremental pipeline for freshness. This architecture plausibly delivers **5–10× token reduction on typical coding tasks and 20×+ on exploration-heavy workflows**, matches sub-100 ms query latency goals, and keeps the entire stack local and single-binary. The single biggest design choice is to stay **symbol-first, not embedding-first** — published evidence (Aider's 26.3% SWE-Bench Lite with only a repo map, Serena's LSP-only approach, SWE-Bench's 2.4× oracle-vs-BM25 gap) shows structural retrieval dominates embeddings for agent workflows, while embeddings earn their keep only as a fallback for fuzzy natural-language queries. Everything below elaborates this stance with citations, schemas, and a phased roadmap.

One important caveat up front: the reference repository `https://github.com/darshil3011/codedrift` **could not be located as of 18 April 2026**. Direct fetches returned permission errors; GitHub search shows `darshil3011` has 20 public repos but no `codedrift`; other `codedrift` namespaces (Peter Mösenthin, Jakob Heuser) are unrelated personal projects. The repo may be private, renamed, deleted, or the URL may contain a typo. Given the author's adjacent work (`AutoMetaRAG`: Python + Qdrant + OpenAI + metadata-filtered hybrid search), codedrift is almost certainly a Python RAG-on-code prototype of that same shape; the report below treats it as that archetype and focuses on what a **production-grade Rust successor to any Python-RAG-over-code MCP** must do better.

## 1. What codedrift most likely is, and why it falls short

Based on the author's other repos and the broader category of early-2025 code-RAG MCPs (codedrift, codebase-rag, various `mcp-code-indexer` projects), codedrift is best modelled as: a Python MCP server that chunks source files with LangChain-style splitters, embeds chunks with an OpenAI or Sentence-Transformers model, stores them in a cloud or local vector DB (Qdrant/Chroma), and exposes a `search_code` tool that returns top-k chunks for natural-language queries. This class of tool shares a consistent set of weaknesses that a production Rust rewrite must fix:

**Indexing is coarse and file-level.** Line-count or char-count chunking cuts functions in half and scatters class context across chunks; the LLM sees half-signatures, unclosed braces, and orphan docstrings. Continue.dev explicitly moved away from this to tree-sitter AST chunking for exactly this reason (blog.continue.dev/accuracy-limits-of-codebase-retrieval). **A Rust rewrite must chunk at symbol boundaries** extracted from tree-sitter's `tags.scm` queries.

**Semantic-only retrieval is the wrong default.** Vector search collapses on short, exact queries (`findUserById`, `CONFIG_TIMEOUT`), and on string-literal references (i18n keys, SQL fragments, Bazel targets). SWE-Bench's own ablation showed BM25 retrieval delivered 1.96% Claude-2 solve rate vs 4.8% oracle (arxiv 2310.06770), and Aider's repo-map-only approach found the right file **70.3%** of the time on SWE-Bench Lite (aider.chat/2024/05/22/swe-bench-lite.html) — strictly better than BM25 and directly comparable to embedding-based Cody. **The Rust rewrite must be structural-first, semantic-as-fallback.**

**The index goes stale.** Python RAG prototypes typically re-index on an interval or on explicit command. Agents then read reality from a snapshot five minutes behind `HEAD`. Cursor's Merkle+embedding-cache approach (cursor.com/blog/secure-codebase-indexing) is the correct pattern: hash every file, bubble hashes to root, recompute only diverged subtrees on every save. **A Rust rewrite with `notify` + blake3 + `gix` can keep the index <1 s behind a file save.**

**Scale ceiling is low.** Python + remote vector DB + per-chunk embedding API calls break economically around the million-LOC mark. The Rust-native competitor `codebase-memory-mcp` (DeusData) reports indexing the 28M-LOC Linux kernel in three minutes; `narsil-mcp` covers 32 languages with a single static binary. **The Rust rewrite should target monorepo-scale out of the gate.**

**Tool surface is generic and tokens-expensive.** A single `search_code(query)` returns raw text chunks; the agent still has to call Read to fetch surrounding context. Serena's multi-tool surface (`find_symbol`, `find_referencing_symbols`, `get_symbols_overview`, `search_for_pattern`) — **26 tools in ~24K schema tokens**, recently reduced to 18 tools / 12K tokens in `serena-slim` — demonstrates the design direction: many narrow tools beat one wide one, but schema size now matters enough that Anthropic added automatic tool-search in Claude Code Sonnet 4+ (platform.claude.com/docs/en/agents-and-tools/tool-use/tool-search-tool), cutting a 77K-token toolset to 8.7K upfront via BM25-over-tool-descriptions.

**Python async loops contaminate subagents.** Serena spent months removing the asyncio-forked-process isolation it needed to survive inside MCP hosts. **Rust + Tokio sidesteps this entire class of bug.**

## 2. The code-intelligence landscape, mapped to Rust reusability

The design space has converged on five archetypes; each has a clear Rust port path.

**Aider's repo-map (tree-sitter + personalized PageRank)** is the single most reusable design. Aider extracts tag tuples `(rel_fname, fname, line, name, kind ∈ {def,ref})` via tree-sitter `.scm` queries (fallback to Pygments for languages where only defs exist), builds a NetworkX `MultiDiGraph` with edge-weight heuristics that **10× identifiers mentioned in chat, 10× long mixed-case identifiers, 0.1× `_private` and >5-file-common names, 50× references from chat-added files**, and runs personalized PageRank with a `sqrt(num_refs)` multiplier (deepwiki.com/Aider-AI/aider/4.1). A binary search then packs the top-ranked tags into `max_map_tokens` (default 1024, expanded 8× when no files are in chat), stopping within 15% of target. The rendering uses `grep_ast`'s `TreeContext` with colors off, no line numbers, zero padding. Aider proves this design scores **SOTA 26.3% on SWE-Bench Lite** with no RAG (aider.chat/2024/05/22/swe-bench-lite.html). Rust port: `tree-sitter` + `petgraph` + `tiktoken-rs` + a Rust equivalent of `TreeContext` (the single hardest piece, ~500 LOC); the `.scm` files are MIT-licensed and reusable verbatim.

**Cursor's Merkle + embedding + Turbopuffer** design is the reference for **freshness and index reuse**. Leaves = SHA-256 of file content, inner nodes = hashes of children; a 50k-file workspace produces ~3.2 MB of hash metadata. Embeddings are keyed by chunk content hash, so unchanged code is never re-embedded; simhashes over team members' trees enable the 7.87 s → 525 ms median onboarding Cursor published (cursor.com/blog/secure-codebase-indexing, 27 Jan 2026). Cursor's claimed agent-accuracy lift from adding semantic search: **+12.5%**. For a local MCP the Merkle and cache keying translate directly; the team-reuse feature is optional.

**Sourcegraph SCIP** is a production-grade static format — Protobuf, ~4× smaller than gzip'd LSIF, human-readable symbol monikers (`rust-analyzer cargo ide 0.0.0 moniker/def_to_moniker().`). Rust-analyzer emits SCIP natively via `rust-analyzer scip <path>`; `scip-python`, `scip-typescript`, `scip-java`, `scip-clang`, `scip-ruby`, `scip-dotnet` cover the rest. The official **`scip` Rust crate (v0.6.1)** parses the Protobuf and the moniker strings (docs.rs/scip). **Recommendation:** use SCIP as the optional "precision tier" — run an indexer out-of-process, mmap the resulting index, serve `goto_definition`/`find_references`/`workspace_symbol` from it with O(1) lookups. Known bugs (rust-analyzer PR #17988: refs mis-classified as defs until Aug 2024; PR #13456: local crates demoted to local-only) mean SCIP still isn't a drop-in oracle, but it is the best static format available.

**Serena's LSP delegation** is semantically perfect and operationally painful. Rust-analyzer can take 5–10 min to warm up on big Cargo workspaces and occupies 1 GB idle, 12–40 GB on monorepos (rust-lang/rust-analyzer#20917, #20818; users.rust-lang.org/t/133914). jdtls needs JVM heap; clangd needs `compile_commands.json`; gopls needs `go.mod`. Serena's 40+ language claim is largely "it spawns whichever LSP you have installed" — setup friction is the real ceiling. For a Rust MCP, the reusable pattern is: `async-lsp` + `ls-types` (active fork of the unmaintained `lsp-types`) as client, per-language lazy-spawned pools, cache `workspace/symbol`/`documentSymbol`/`references` by `(path, mtime)`, translate payloads to compact `path:line:col` triples rather than full LSP `Location` objects. **Treat LSP as a tier-3 oracle, not the primary backend.**

**Zed's `semantic_index`** (Rust, in-tree) is the cleanest open-source code-embedding pipeline to read: tree-sitter symbol objects, optional body-collapsing for nested classes, **token-count batching** (not span-count) for embedding API throughput, content-hash cache that cut incremental re-indexing tokens by **~75%** (zed.dev/blog/zed-weekly-18). Model/dim/chunker undisclosed; likely OpenAI `text-embedding-3-small`. The Rust-native code is directly readable for patterns; replace the OpenAI call with a local ONNX model via `fastembed` + `ort` for privacy and zero-per-query cost.

**The smaller primitives** — `ast-grep` for structural search (`ast-grep-core` is on crates.io 0.42 with 1.18M downloads but its Rust API is officially unstable; the CLI-as-subprocess path is safer); universal-ctags / GNU global (regex parsers, no error-tolerance, no reason to use them when tree-sitter is available — Aider switched away from ctags for exactly these reasons); Repomix/gitingest for whole-repo compression (useful only for small repos < 100K tokens or for orientation prompts) — are all worth knowing but none earn primary-engine status.

**Rust-side storage/search/vector candidates.** Tantivy 0.25 is the undisputed winner for BM25 (~300 GB/hr indexing, 2× Lucene query latency, <10 ms startup); single-writer / many-reader model suits an indexer/server split. LanceDB is the best embedded vector store for code intel because it co-locates SQL-style scalar filters with vector search (`WHERE lang = 'rust' AND path LIKE 'crates/%' ORDER BY vector <-> $q`); usearch is the minimal-footprint alternative when you don't need SQL. For the canonical KV/metadata layer, rusqlite with `bundled` + FTS5 trigram tokenizer covers prefix/fuzzy/symbol lookup with battle-tested crash safety; **redb 2.3+ is the pure-Rust alternative** if clang-free builds matter, and sled should be avoided for new projects. Notify 8.2 + notify-debouncer-full 0.6 is the only real file-watching option; always debounce (~250 ms) and always fall back on FSEvents coalescing / Windows buffer overflow / inotify `max_user_watches` exhaustion. The `ignore` crate (ripgrep's) owns fast gitignore-aware traversal; `gix` (gitoxide) owns blob/tree access and outperforms `git2` on large repos (starship migrated for this reason).

**Local code embeddings.** For fallback semantic search: **`jina-embeddings-v2-base-code`** (161M params, 768-d, 8192 ctx, 30 languages, Apache-2.0) via `fastembed-rs` on top of `ort 2.0` ONNX Runtime. It runs under 300 MB RAM, CPU-acceptable (batched 32–64), and is purpose-built for code. `nomic-embed-code` (7B, 3584-d) is SOTA but only practical with a GPU. `candle` is the pure-Rust alternative; reports suggest 3–4× slower GPU perf than PyTorch, so use `ort` by default.

## 3. Core design decisions — opinionated recommendations

**Indexing strategy: symbol-first hybrid with three tiers, not embedding-first.** Tier 1 (always on) = tree-sitter tag extraction + PageRank repo-map. Tier 2 (on demand, cached) = Tantivy BM25 over symbols and content with a camelCase/snake_case-splitting tokenizer. Tier 3 (optional) = local ONNX embeddings into LanceDB for fuzzy natural-language queries ("where do we validate email?"). Tier 4 (power user) = SCIP index generated by the language's native indexer for cross-repo precision, or live LSP for rename/edit tasks. The ordering matters: published SWE-Bench and Aider numbers show structural beats semantic for the agent's primary failure mode of "which file has the bug?"; semantic earns its place only on the long tail.

**Chunking strategy: symbol-level, never line-count.** One logical unit per symbol: function, method, class (as a collapsed overview + child symbols), top-level const/type, module docstring. Store `(path, symbol_name, kind, byte_start, byte_end, line_start, line_end, parent_symbol, signature, doc, body_hash)`. For embedding, feed `{language} {kind} {qualified_name}({signature}) {doc}\n{body}` truncated to the model's context window. Body-hash the content-addressable key lets you reuse embeddings across copies and across refactors that don't touch bodies — Zed's trick, confirmed 75% token savings on re-index.

**Storage backend: SQLite + Tantivy + LanceDB, not one mega-store.** rusqlite 0.37 with `bundled` feature and `PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL;` is the canonical symbol table (write-concurrent enough for single-indexer, crash-safe, one file to back up, FTS5 trigram for fuzzy). Tantivy is the BM25/positional engine — delete-by-term + re-insert on updates, commit every 2 s or every 500 docs. LanceDB holds vectors and enables scalar filtering in the same query. Do **not** try to shove everything into one engine: Tantivy's primary-key lookups are slower than SQLite's, LanceDB's full-text is weaker than Tantivy's, SQLite's vector extensions are less mature than LanceDB. The schema sketch:

```sql
CREATE TABLE files (
  path       TEXT PRIMARY KEY, lang TEXT NOT NULL, size_bytes INTEGER,
  content_hash BLOB NOT NULL, mtime INTEGER NOT NULL, indexed_at INTEGER);
CREATE TABLE symbols (
  id INTEGER PRIMARY KEY, path TEXT REFERENCES files(path) ON DELETE CASCADE,
  name TEXT NOT NULL, qualified_name TEXT NOT NULL, kind TEXT NOT NULL,
  signature TEXT, doc TEXT, parent_id INTEGER REFERENCES symbols(id),
  byte_start INTEGER, byte_end INTEGER, line_start INTEGER, line_end INTEGER,
  body_hash BLOB NOT NULL);
CREATE INDEX symbols_name ON symbols(name COLLATE NOCASE);
CREATE INDEX symbols_qname ON symbols(qualified_name COLLATE NOCASE);
CREATE VIRTUAL TABLE symbols_fts USING fts5(
  name, qualified_name, signature, doc,
  content='symbols', content_rowid='id', tokenize='trigram');
CREATE TABLE refs (                                    -- Aider-style edges
  src_path TEXT, src_symbol_id INTEGER,
  ident TEXT NOT NULL, dst_symbol_id INTEGER, weight REAL);
CREATE INDEX refs_ident ON refs(ident);
```

**Search/retrieval: hybrid, with symbol search as the default.** For an exact identifier query, hit the `symbols` table by name. For a fuzzy identifier, fall through to `symbols_fts` (trigram). For natural-language, fall through to LanceDB vector search over symbol embeddings with language+path filters. For a regex/textual grep, use Tantivy phrase queries over bodies, or shell out to `ripgrep` when the pattern is regex-heavy. Rank across modes by reciprocal-rank fusion (RRF) when returning unified results.

**Incremental indexing: notify + Merkle + git.** A `notify-debouncer-full` watcher runs at 250 ms debounce over the repo root; each batch of events filters through an `ignore::gitignore::Gitignore` matcher; surviving paths are hashed with blake3, bubbled into a per-directory Merkle tree, and diffed against the previous snapshot. Changed files are re-parsed with a pooled tree-sitter parser (one parser per thread, via `crossbeam_queue::ArrayQueue`), symbols are diffed at the body-hash level, and only novel bodies are re-embedded. Boot path uses `gix` to diff `HEAD` against the last indexed commit to avoid full rescan. On notify `Error::Overflow` or FSEvents coalescing, fall back to `git status -s` + targeted rehash. **Target: file save → queryable in <1 s on repos up to 500K LOC; <5 s on million-LOC monorepos.**

**Language support strategy: bundle 9 grammars at build time, add LSP fallback.** Compile in `tree-sitter-{rust, python, typescript, tsx, javascript, go, java, c, cpp}` via their build.rs, plus `tree-sitter-tags` 0.25 for the symbol extraction pass. Use **nvim-treesitter and Aider's `.scm` files as the query source** (both permissively licensed, maintained, convergent). For niche languages (Kotlin, Swift, Ruby, PHP, Zig, Nix, Haskell, …) either pull tree-sitter grammars dynamically via the wasm runtime (`tree-sitter`'s `wasm` feature + wasmtime) or delegate to the matching LSP via `async-lsp`. Ship a `--language <lang>=<lsp-cmd>` config hook so users can register their own LSPs without recompiling.

**Token-efficient response formats: five disciplined shapes.** (1) Skeleton view — signatures + docstrings + nested symbol names, bodies elided with `...`, one or two blank lines between top-level symbols; Aider-style. (2) Outline view — bullet tree of `{kind} {name} [line]` only, no signatures. (3) Symbol card — one symbol's full source plus its parent signature chain and its direct callers/callees as `path:line` triples. (4) Reference digest — count + top-10 call sites as `path:line` triples, not full snippets; the agent asks for expansion if it needs it. (5) Compact diagnostics — `file:line:col: message` like `cargo check`. **Rule:** every tool emits one of these five shapes, never free-form text. This is what lets the server ship a meaningful `description` field under the 2 KB Claude Code truncation limit.

**MCP tool design: eight tools, all prefixed, all narrow.** The replacement set:

| Tool | Replaces | Output shape |
|---|---|---|
| `code_outline(path)` | Read (orientation) | Outline view — one line per top-level symbol |
| `code_skeleton(path, depth=2)` | Read (broad) | Skeleton view — signatures + docstrings |
| `code_read_symbol(path, symbol \| line)` | Read (targeted) | Symbol card — full body + parents + callers digest |
| `code_find_symbol(name, kind?, lang?)` | Grep (symbols) | Array of `{qualified_name, path, line, kind, signature}` |
| `code_find_refs(symbol)` | Grep (usages) | Reference digest — count + top-k |
| `code_search(query, mode="hybrid"\|"exact"\|"regex"\|"semantic")` | Grep (freeform) | Array of `{path, line_start, line_end, snippet, score}` |
| `code_list(glob?, lang?)` | Glob | Array of `{path, lang, loc, symbol_count, top_symbols[3]}` |
| `code_repo_map(focus_files?, token_budget=2000)` | — (new) | Skeleton view of top-PageRank symbols under budget |

Every tool name is prefixed `code_` to avoid collisions with Claude Code's priors for `Read`/`Grep`/`Glob`. Descriptions must (a) be under 2 KB each, (b) state the output shape and expected token size, (c) list representative inputs, (d) explicitly say "prefer over Read/Grep/Glob when …". Anthropic's tool-search BM25 ranks these descriptions, so **specific verbs and distinctive vocabulary ("outline", "skeleton", "symbol card", "reference digest") are more important than polished prose.** A project-level `CLAUDE.md` rule ("Use `code_*` tools before Read/Grep/Glob") further nudges adoption, as `token-savior` documents.

## 4. Token economics — what the numbers actually say

The conservative, evidence-anchored estimate: **5–10× total-token reduction on typical coding tasks, 20×+ on exploration-heavy sessions**, with the bulk of savings coming from eliminating redundant Reads rather than smarter Greps. The strongest data points, in order of rigor:

**SWE-Bench (arxiv 2310.06770)** is the only academic retrieval-ablation with numbers: Claude 2 solved 1.96% with BM25 @ 13K tokens vs 4.8% with oracle file retrieval — a **2.4× quality uplift** from better retrieval, with BM25 missing all gold files in ~50% of instances and over-retrieving in another 40%. Aider's repo-map approach hit the correct file **70.3%** of the time on SWE-Bench Lite without RAG. This is the ground truth: **structural retrieval beats BM25 and is competitive with embeddings for agent tasks.**

**grepai's controlled benchmark** (yoanbernabeu.github.io/grepai/blog/benchmark-grepai-vs-grep-claude-code) on the Excalidraw codebase (155K LOC TS, 5 identical questions × 5 runs) is the cleanest apples-to-apples measurement: tool calls 139 → 62 (-55%), fresh input tokens 51,147 → 1,326 (**-97%**), cache-creation tokens 563,883 → 162,289 (-71%), subagents launched 5 → 0, Glob calls 13 → 0, total billed cost $6.78 → $4.92 (**-27.5%**). The gap between -97% fresh-input and -27.5% billed cost tells the real story: **Anthropic's prompt caching already absorbs most of the obvious savings, so the honest win is on fresh input and subagent elimination**, not on total $.

**Zilliz/claude-context** reports ~40% token reduction at equal recall (zilliz.com/blog). **mixedbread mgrep** reports ~2× token reduction on a 50-task benchmark. **Serena** self-reports 70–80% token savings anecdotally. **token-savior** claims 97% character reduction (marketing, unverified). **Speakeasy Gram's Dynamic Toolsets** reported 96% input-token reduction from tool-schema compression — an orthogonal axis entirely, and a reminder that **the MCP schema itself can eat 55K+ tokens before the conversation starts** (Anthropic's own figure for a 58-tool / 5-server setup).

**Anthropic's own Nov 2025 "code execution with MCP"** example documents a 150K → 2K token workflow (98.7% reduction) by letting the model write code that calls MCP tools instead of chaining tool-calls directly — relevant because a well-designed Rust MCP server can expose a `run_query` or `run_analysis` tool that bundles multiple internal lookups into one round trip, collapsing round-trip overhead.

**Numerical model for this server's expected savings**, assumptions validated against the sources above: average source file ~300 LOC / 3,500 tokens; skeleton ~30 LOC / 300 tokens (10× compression per file, consistent with Aider's published map example and Continue.dev's top-level-extraction fallback); typical task touches 3–7 gold files but current Claude Code explores 10–30 (grepai showed 43 Reads for 5 questions). Baseline per-task exploration: 15 files × 3,500 = ~52K tokens plus grep noise = **75–120K tokens on exploration alone**. With `code_repo_map` (1–4K) + 3 targeted `code_read_symbol` calls (3 × ~800 = 2,400) + 2 `code_find_refs` (~1K total) = **~6–8K tokens**, i.e. a **~10× exploration reduction**, which lines up with grepai's -97% fresh-input figure minus the cache-absorbed component. Savings dominate when (a) the task is exploratory, (b) the repo is large enough that whole-file reads dominate, (c) the agent would otherwise launch subagents for exploration.

**Failure modes to mitigate.** Skeleton views hide implementation-level bugs (off-by-one, wrong null check) — mitigate by always letting the agent upgrade from skeleton to full body via `code_read_symbol`. Symbol search misses string-literal references (SQL, i18n keys, DI string targets, Django settings) — mitigate by keeping `code_search(mode="regex")` as a first-class tool and not deprecating text search. Embedding search collapses on short/exact queries — mitigate by making `mode="hybrid"` route exact-identifier queries to symbol lookup before vectors. Stale indexes — mitigate by exposing `indexed_at` on every tool response so the agent can detect drift. Tool-schema bloat — mitigate by keeping the tool count at 8 and descriptions under 2 KB each; lean on Claude Code's built-in tool search to defer schemas.

## 5. Rust implementation specifics

The production stack, pinned to versions verified on crates.io April 2026: **rmcp 1.3** (official MCP SDK, 4.7M+ downloads, weekly releases, endorsed by Codex/Anthropic/rustic-ai), **tokio 1.49** with `rt-multi-thread`, **tree-sitter 0.25** (pinned to language-crate ABI), **tree-sitter-tags 0.25**, **tantivy 0.25** with a custom `CodeTokenizer` implementing the `tantivy-tokenizer-api` 0.7 trait, **rusqlite 0.37** with `bundled + blob + array + modern_sqlite` features, **lancedb 0.23**, **fastembed 5** + **ort 2.0-rc.11** (despite the `rc` label the maintainers explicitly declare it production-ready), **notify 8.2** + **notify-debouncer-full 0.6**, **ignore 0.4** (ripgrep's), **gix 0.70** default-features-off with `blob-diff` + `worktree-stream`, **clap 4.5** + **clap_complete**, **tracing 0.1.43** + **tracing-subscriber 0.3.22** with `env-filter + fmt + json`, and for hot paths **smallvec 1**, **compact_str 0.9**, **bumpalo 3**, **ahash 0.8**, **blake3 1**, **crossbeam-queue 0.3**, **arc-swap 1**, **rayon 1**. Skip sled (beta-forever), skip `git2` unless you hit a specific gap in `gix`, skip `async-std`/`smol`, skip Qdrant for an embedded server, and treat `ast-grep-core` as subprocess-only until its API stabilizes.

**The non-obvious pitfall is tree-sitter threading.** `Parser` is `Send` but **not `Sync`**; `Node<'tree>` is `Send` but **not `Sync`** and borrows from the tree, so holding a `&Node` across an `.await` breaks in multi-threaded runtimes (tree-sitter#1241). The fix is a bounded parser pool per language and strict discipline against carrying nodes across await points — extract byte ranges and owned strings first. `Tree` itself is both `Send + Sync` (immutable). The canonical pool pattern:

```rust
pub struct ParserPool {
    lang: tree_sitter::Language,
    pool: crossbeam_queue::ArrayQueue<tree_sitter::Parser>,
}
impl ParserPool {
    pub fn new(lang: tree_sitter::Language, cap: usize) -> Self {
        Self { lang, pool: crossbeam_queue::ArrayQueue::new(cap) }
    }
    pub fn with<R>(&self, f: impl FnOnce(&mut tree_sitter::Parser) -> R) -> R {
        let mut p = self.pool.pop().unwrap_or_else(|| {
            let mut p = tree_sitter::Parser::new();
            p.set_language(&self.lang).expect("lang");
            p
        });
        let r = f(&mut p);
        let _ = self.pool.push(p);
        r
    }
}
```

**Tantivy's single-writer model** means exactly one `IndexWriter` per on-disk index (enforced by lockfile), with many `IndexReader`s. Commit every 2 s or 500 docs in the indexer task; MCP query handlers reload readers via `ReloadPolicy::OnCommit`. Wrap every call to tantivy from async code in `spawn_blocking` — it is a sync library and blocking the reactor on a 20 ms fetch will surface immediately at load.

**Other hot-path notes.** Embedding inference must always batch (32–64); single-item CPU latency is ~the same as batch-32 per-item latency because transformers are memory-bandwidth bound. Memory-mapping: on Linux watch `vm.max_map_count` for multi-segment Tantivy indexes; on Windows mmapped files can't be renamed while open. Avoid allocation in query hot paths with `bumpalo` arenas for per-request AST scratch, `compact_str` for symbol names (inlines strings ≤ 24 bytes), `smallvec` for the small-fanout fields that otherwise go to the heap. Every tool response should stream through `serde_json` with minimal intermediate allocation. **On stdio transport, configure `tracing_subscriber::fmt().with_writer(std::io::stderr)` — a stray `println!` anywhere corrupts the JSON-RPC channel** and is the #1 MCP server bug category (postman/mcp-server shipped broken for exactly this reason).

**Workspace layout** — eight crates in a Cargo workspace with `resolver = "2"`, `edition = 2021`, `rust-version = 1.85`, and a `[workspace.dependencies]` section that version-unifies every dep:

```
crates/
  ci-core/       # Symbol, Span, Language, errors; serde only
  ci-parser/     # tree-sitter pools, tags.scm loading, symbol extraction
  ci-storage/    # rusqlite schema + migrations, blake3 content hashing
  ci-search/     # tantivy schema + CodeTokenizer + query builder
  ci-embed/      # fastembed + lancedb; batcher + content-hash cache
  ci-indexer/    # Merkle walker + notify loop + gix diff
  ci-server/     # rmcp ServerHandler + the 8 tool impls
  ci-cli/        # clap binary: code-intel {serve|index|query|reindex}
xtask/           # release, bench, schema-dump
```

`ci-core` has no heavy deps and is depended on by everything else. `ci-server` talks to storage/search/embed through traits in `ci-core`, so swapping LanceDB for usearch later is a one-module change. `ci-cli` is the only binary; release via `cargo-dist` for prebuilt artifacts.

**Large-monorepo handling.** Shard Tantivy indexes per language (`indexes/rust/`, `indexes/ts/`); a `rust`-only query then never touches TS posting lists. One embedding per symbol, not per file — a 1 MB source file becomes ~50 precise embeddings, not one blurry one. Incremental re-index is keyed on `(path, blake3(content))` stored in SQLite; unchanged files skip parse + embed entirely. Use `arc_swap::ArcSwap<IndexReader>` to hot-swap tantivy readers on commit without blocking queries. For the initial cold index, drive parsing through `rayon::par_iter` over the `ignore::WalkBuilder` output.

## 6. Production concerns — concurrency, observability, security, testing

**Concurrency.** MCP hosts commonly dispatch multiple simultaneous tool calls; the server must be fully parallel on reads. Architecture: one background indexer task (owns the tantivy `IndexWriter` and SQLite write connection), a read-only SQLite connection pool (`r2d2_sqlite`), shared `Arc<IndexReader>` for tantivy, shared `Arc<LanceConnection>` for vectors, and per-request tokio tasks for tool handlers. CPU-bound work (tree-sitter parse, embedding inference) goes through `spawn_blocking` or a dedicated `rayon` pool. The indexer task receives change events through a bounded `tokio::sync::mpsc` channel — backpressure when the watcher bursts is explicit.

**Error handling and graceful degradation.** Every tool returns a `Result<CallToolResult, ErrorData>`; on recoverable errors (stale index, missing file, parse error) return `isError: true` in the tool result per SEP-1303 so the model can self-correct, not a JSON-RPC protocol error. On unrecoverable errors (storage corruption) log at ERROR and mark the server in a degraded state that still allows raw-file reads. Always degrade cleanly: if the tantivy reader fails, fall back to SQLite FTS5; if embeddings are unavailable, route `mode="semantic"` to hybrid; if a language grammar fails, emit an outline from ctags-like line patterns.

**Observability.** `tracing` spans on every tool handler with `tool_name`, `repo`, `token_count_out`, `latency_ms`, `cache_hit` fields. `tracing-subscriber` with JSON formatter and a file appender (never stdout under stdio transport). Expose `/metrics` Prometheus endpoint under Streamable HTTP; counters for `tool_calls_total{tool,status}`, histograms for `tool_latency_seconds{tool}`, gauges for `index_size_bytes`, `symbols_total`, `last_indexed_timestamp`. Optional `opentelemetry 0.31` + `tracing-opentelemetry 0.32` bridge for OTLP export. Add a `code_stats` MCP tool that returns index health — the agent itself can then detect staleness and request `code_reindex`.

**Configuration.** Per-repo config at `<repo>/.code-intel/config.toml`: ignore patterns (over and above `.gitignore`), language grammar overrides, LSP command hooks, embedding model choice, index size caps. Global config at `~/.config/code-intel/config.toml`. Env vars (`CODE_INTEL_LOG`, `CODE_INTEL_INDEX_DIR`, `CODE_INTEL_EMBED_MODEL`) override config.

**Generated code / vendored deps / build artifacts.** Default ignore set: `node_modules`, `target`, `dist`, `build`, `.next`, `.venv`, `__pycache__`, `vendor`, `third_party`, `generated`, `.git`, lockfiles, `*.min.js`, `*.bundle.js`, `*.pb.go`, and any `*.generated.*`. All through `ignore::gitignore::Gitignore` with a builtin fallback ignore file. **Detect generated files** by scanning the first 8 lines for markers like `DO NOT EDIT`, `@generated`, `// Code generated by`, `AUTO-GENERATED`. Size caps: default 1 MB per file, 100 KB for the embed pipeline (truncate, don't drop).

**Security.** Validate every `path` argument with `path.canonicalize()` against the resolved repo root; reject any path that escapes (`../`, symlinks out of root). Strip secrets before returning content — run a fast regex set (AWS keys, private-key headers, OpenAI-style keys) or integrate `gitleaks`-style rules on every response that includes file content. Refuse to return paths matching `.env*`, `*.pem`, `*.key`, `id_rsa*` by default; gate with an explicit `allow_sensitive: true` flag. Because this server runs with full repo read access, run it under the same user as the agent with a `umask` that prevents world-write on the index directory. Never execute code; never follow HTTP redirects from embedded URLs; never fetch remote resources implicitly. For shared-index multi-repo mode, isolate per-repo indexes in separate directories with distinct SQLite files — no cross-repo joins.

**Index migration.** Embed a schema version integer in both the SQLite `meta` table and the tantivy `Index.meta.json`. On server startup, compare against the binary's compile-time version; if older, run `rusqlite_migration` to upgrade the SQL schema, and rebuild the tantivy index from scratch when its schema changed (tantivy doesn't support in-place schema migration — this is cheap enough for code indexes). Expose `--force-reindex` and `code_reindex` MCP tool for manual recovery.

**Multi-repo.** First-class design, not a feature flag. MCP hosts pass `root_uri` in `initialize`; the server maintains a `DashMap<PathBuf, RepoState>` with each `RepoState` owning its own SQLite/Tantivy/LanceDB/ParserPools and its own Merkle snapshot. A `code_activate_repo(path)` tool switches the active repo for stateful tools; path-qualified tools (`code_find_symbol`) can take an explicit `repo:` prefix. Laziness matters: don't index all mounted repos on startup — only those the agent touches.

**Testing strategy.** Three layers. Unit tests for each parser/tokenizer/storage module against small fixtures (one `.rs` file with 5 known symbols, one `.py` with known refs). Property tests (`proptest`) for the Merkle diff: generate random file-trees, apply random edits, assert the diff visits only touched paths. Integration tests on real OSS repos checked in as git submodules: **Aider itself, ripgrep, tokio, axum, and a Python project (e.g., FastAPI)**; run each of the 8 MCP tools against known ground truth (hand-authored YAML of expected symbols/refs). End-to-end tests against a mocked MCP client: assert tool descriptions fit 2 KB, assert tool responses parse as the documented shape, assert stdio transport never writes to stdout except JSON-RPC (test by running the binary and scanning stdout for non-JSON). Benchmark harness with `criterion` on (parse Rust file, symbol lookup, repo-map generation, embed 1000 symbols) to catch regressions; run nightly against the SWE-Bench Lite retrieval task to catch quality drift. Fuzz the MCP request handler with `cargo-fuzz` against the JSON-RPC payload shape.

## 7. MCP protocol state — April 2026 specifics

The current spec is **2025-11-25** (modelcontextprotocol.io/specification/2025-11-25). Anthropic donated MCP to the Linux Foundation's Agentic AI Foundation in December 2025 (co-founded with Block and OpenAI); the 9 March 2026 roadmap post confirms no new spec version has cut since November. **Streamable HTTP** is the current recommended remote transport — single endpoint handling POST and GET with optional SSE upgrade — and it **deprecated the old two-endpoint HTTP+SSE transport** in the 2025-03-26 spec. Atlassian and other vendors are sunsetting SSE endpoints by 30 June 2026. **Stdio remains the default for local servers** and is unchanged. **rmcp** is the official Rust SDK; latest version at time of research is 1.5.0 (released ~1 day prior), 1.0.0 was cut in early March 2026, and 4.7M+ downloads on crates.io — the Codex team migrated from their custom `mcp-types` crate to rmcp in early 2026 and rustic-ai's codeprism post-mortem explicitly advises "Don't build your own. Start with rmcp."

**Tool-search changes the design.** As of Sonnet 4+ / Opus 4+ in Claude Code, tool definitions are **deferred by default** — the model sees tool names only until it invokes `tool_search` with a BM25 query over tool names/descriptions/argument descriptions (platform.claude.com/docs/en/agents-and-tools/tool-use/tool-search-tool). Anthropic's own benchmark: 50+ MCP tools cost 77K tokens upfront without tool search vs 8.7K with it — **85% upfront reduction**. Implications for the tool design: (a) distinctive vocabulary in names and descriptions matters more than prose polish because BM25 decides inclusion, (b) under-2 KB descriptions are enforced (silent truncation above), (c) the penalty for exposing 20 narrow tools is now small, so the eight-tool surface proposed above is comfortable. Still prefix everything with `code_` to disambiguate from built-in `Read`/`Grep`/`Glob` that enjoy strong model priors.

**Claude Code gotchas to guard against.** (1) **stdout pollution** — any `println!`, any library banner, any default-stderr-to-stdout logger breaks the connection silently. Guard with a startup assertion that no write to stdout happened before `ServiceExt::serve(stdio())`. (2) **Tool name collisions** across servers — MCP doesn't globally namespace; use `code_*` prefixes. (3) **`ANTHROPIC_BASE_URL` pointing at a proxy disables tool search silently** — document `ENABLE_TOOL_SEARCH=1` as a recommended env. (4) **SSE-first clients** (OpenCode #6242) still try the deprecated SSE transport first; advertise Streamable HTTP explicitly in your capabilities manifest. (5) **Tool descriptions truncate at 2 KB**; examples at the end are the first to disappear. (6) **Server instructions** (the `instructions` field) also truncate; put CLAUDE.md-style tool-preference rules there. (7) **`Read` and `Grep` are first-class Claude Code tools with model priors** — users must opt out explicitly or the agent will keep reaching for them; document a recommended `.claude/settings.json` hook that disables them when the MCP server is active (token-savior uses exactly this pattern).

## 8. The competitive landscape and the differentiation opportunity

Direct competitors fall into five groups. **Python LSP wrappers** — Serena leads the category (9.5K+ GitHub stars, 40+ languages, rich tool surface) but inherits LSP startup cost, memory, setup fragility, and Python async overhead. **Python RAG-over-code** — codedrift (if it exists), Code-Index-MCP (johnhuang316), claude-context (zilliztech, TS/Node, requires Milvus + embedding API, Node 20–24 pinning) — strong on familiarity, weak on freshness, scale, and local-only operation. **Go servers** — mcp-language-server (isaacphi, LSP delegation, minimal, single LSP per instance), grepai (Ollama-backed semantic grep, call-graph tracing, single binary), Code-Index-MCP (trondhindenes, Zoekt trigram) — good engineering, but single-axis (semantic-only or trigram-only). **Structural MCPs** — token-savior (Python, 69 tools, SQLite+FTS5+vectors, self-reported 97% reduction) — closest philosophical match to the proposed design but Python-delivered and tool-count-heavy. **Rust-native competitors** — **codebase-memory-mcp (DeusData)** is the real direct competitor: tree-sitter across 66 languages, single static binary, claims Linux kernel (28M LOC) indexed in 3 minutes, sub-ms queries, 14 MCP tools including `get_architecture`; and **narsil-mcp (postrv)** with 90 tools, 32 languages, OWASP/CWE security rules, optional Oxigraph RDF graph, neural semantic search via Voyage/OpenAI, type inference. Also **codeprism (rustic-ai)** using rmcp.

**The differentiation opportunity** for the proposed server, given that Rust + tree-sitter + single-binary is already a populated category: (1) **Aider-style personalized PageRank repo-map as a first-class MCP tool** — no competitor exposes this; it's the highest-signal-per-token context primitive known. (2) **A principled five-output-shape discipline** — every tool returns exactly one of {outline, skeleton, symbol card, reference digest, diagnostics}, explicitly designed for LLM consumption rather than human dashboards; most competitors leak LSP payloads or raw chunks. (3) **Fully local embeddings via `jina-embeddings-v2-base-code` + `ort`** — claude-context forces a cloud embedding API, Serena has no embeddings, codebase-memory-mcp appears structural-only. (4) **Sub-1-second freshness via `notify` + Merkle + `gix`** — most competitors re-index on interval or explicit command; Cursor's Merkle approach shows this is the correct path. (5) **SCIP-tier precision mode** — nobody in the MCP space consumes SCIP natively, yet rust-analyzer and friends already emit it with `scip` crate parsers available. (6) **Tool-search-aware description design** — most MCPs still optimize for human-readable prose; optimize for BM25 recall instead. (7) **Claude Code `Read`/`Grep`/`Glob` replacement as a complete package**, including the `.claude/settings.json` hook to disable built-ins, not as a parallel option. Ship all seven and the project is the best answer in the space.

## 9. Concrete architectural proposal and phased roadmap

**Architecture (one picture, in words).** The binary, `code-intel`, runs one stdio MCP server process rooted at a repo. On start it resolves the repo root via `gix`, loads `.code-intel/config.toml`, spawns the indexer task, opens the three storage backends (SQLite, Tantivy, LanceDB), and serves rmcp. The **indexer task** owns the write side: `notify-debouncer-full` → `ignore` filter → blake3 hash → Merkle bubble → diff vs previous snapshot → pooled tree-sitter parse on changed files → tag extraction via `tree-sitter-tags` → symbol diff by body-hash → SQLite upsert + Tantivy delete-by-term + add + LanceDB upsert for novel bodies → commit every 2 s. The **server task** owns the read side: 8 rmcp tools, each a pure function of `(repo_state, request)` → one of five output shapes, each backed by SQLite → Tantivy → LanceDB fallback as appropriate. Cross-cutting: `tracing` JSON to stderr + optional OTLP, a `/metrics` endpoint under Streamable HTTP mode, a token-counter (`tiktoken-rs`) that tags every response with its output token count so the agent and the operator can both observe efficiency.

**Data flow for a representative query** (`code_find_refs(symbol="my_crate::Foo::bar")`): rmcp deserializes → server handler resolves the symbol row in SQLite → fetches Tantivy hits on `refs.ident = "bar"` filtered by qualified-name match → loads caller symbol rows in a single SQLite `IN` query → formats Reference Digest (count + top-10 `path:line:col` + a 1-line snippet each) → returns under 400 tokens. Latency budget: SQLite 1 ms, Tantivy 5 ms, SQLite fan-out 3 ms, render 1 ms — **~10 ms p50, under 100 ms p99** even on a million-LOC repo, validated against Tantivy's published 2× Lucene and rusqlite's sub-ms primary-key reads.

**Roadmap — three milestones, each a shippable release.**

**MVP (4–6 weeks, single developer).** Ship the eight-tool rmcp stdio server with tree-sitter-backed symbol extraction for Rust, Python, TypeScript, Go. SQLite + FTS5 trigram only — skip Tantivy and LanceDB. Notify-based re-index on change, full-file blake3 hashing, no Merkle yet. Aider-style repo-map via a pure-Rust `petgraph` + power-iteration PageRank and a hand-rolled skeleton renderer. `code_*` tool names, under-2 KB descriptions, stderr-only logging, path-traversal guards. Benchmark against ripgrep + `Read` on two OSS repos; target **≥5× fresh-input token reduction** on a standard exploration workflow. Release as `code-intel 0.1` via `cargo-dist`; distribute install instructions for Claude Code, Cursor, Windsurf.

**v1 (8–12 weeks).** Add Tantivy with the custom `CodeTokenizer` and the per-language shard layout; migrate FTS5 to Tantivy for full-text, keep FTS5 trigram for symbol fuzzy. Add LanceDB + `fastembed` + `jina-embeddings-v2-base-code` for `mode="semantic"` fallback. Add Merkle-tree snapshots and `gix` cold-start diff. Extend languages to Java, C, C++, JS, TSX. Add `code_repo_map` with personalized PageRank over chat-focused files (takes optional `focus_files` arg). Add streaming HTTP transport with Prometheus metrics. Add the `.claude/settings.json` hook that disables `Read`/`Grep`/`Glob` when the server is active. Target **≥10× exploration-token reduction** against a standardized benchmark suite (SWE-Bench Lite + Excalidraw replay + Aider's own repo). Ship a property-test suite and nightly SWE-Bench retrieval eval.

**Production (quarters, ongoing).** Add SCIP consumption as tier-4 precision mode (run `rust-analyzer scip`, `scip-python`, `scip-typescript` out-of-process; mmap the Protobuf; serve references/definitions from it). Add `async-lsp` live-LSP delegation for rename/edit tools. Multi-repo support with `DashMap<PathBuf, RepoState>`. Wasm-runtime grammar loading for 30+ additional languages. OTLP telemetry, secret-redaction, generated-file detection. Shared-index mode for teams (Cursor-style simhash reuse, optional). Rust SDK for downstream tools to embed the engine as a library. Each quarter, publish a token-economics report on real workloads to validate the 5–10× claim against a growing corpus.

**Success criteria, measured.** (i) Sub-100 ms p99 latency on the eight tools for million-LOC repos, measured by `criterion` and a synthetic MCP client. (ii) Less-than-1-s file-save-to-queryable latency, measured by an E2E harness that writes a known symbol into a watched file and polls `code_find_symbol` until it appears. (iii) 5–10× median-task token reduction vs Claude Code's built-ins on a fixed benchmark suite, measured as Anthropic API-reported `input_tokens` plus `cache_creation_input_tokens`. (iv) SWE-Bench Lite retrieval accuracy ≥ Aider's 70.3% gold-file hit rate, measured monthly. (v) Single static binary under 30 MB (tree-sitter grammars dominate); cold start under 200 ms.

## Conclusion

The evidence points decisively to a symbol-first architecture with embeddings as a fallback, not the other way around — Aider's SWE-Bench result, Serena's adoption, SWE-Bench's 2.4× oracle gap, and grepai's 97% fresh-input reduction all lean in the same direction. The Rust ecosystem has matured enough in 2025–2026 (rmcp 1.3 GA, Tantivy 0.25, LanceDB 0.23, redb 2.3 stable, gix 0.70, notify 8, fastembed 5 on ort 2.0) that a single developer can ship an MVP that credibly competes with Serena and codebase-memory-mcp in six weeks. The real design question is not which vector store or which embedding model, but **what five output shapes the tools emit and how they compose under a 2 KB-per-description / tool-search-ranked regime** — because every token of schema and response is observed by the agent's planner. Get those disciplines right, prefix every tool with `code_`, keep stdout clean, hash everything by content, and the 5–10× token reduction follows from the architecture rather than from heroics. The open territory a production-grade Rust server should claim is the combination nobody has shipped: **Aider's repo-map + Serena's symbol editing + Cursor's Merkle freshness + Zed's content-addressable embedding cache + SCIP's cross-repo precision — all local, all in one binary, all optimized for how Claude Code's planner actually selects and consumes tools in April 2026.**
